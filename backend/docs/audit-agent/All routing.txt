Intent detection – the NLU layer maps the user utterance to one of the high‑level request categories above (Visibility, Meta, Audit, etc.).
Pre‑flight check – if the request needs a specific project ID, the router pulls it from the session; if missing, it returns a “please connect your project” error.
Tool‑selection matrix – each category has a predefined ordered list of tools (the tables above). The router builds a pipeline of calls.
Parallel vs. sequential – most routes run tools sequentially because later steps depend on earlier data. A few low‑cost calls (quickContext, searchMemories) are fired in parallel to reduce latency.
Skill activation – once the required data is gathered, the router injects the appropriate skill (meta-optimizer, schema-generator, etc.) into the LLM context and asks it to produce the final answer.
Response formatting – the final output is passed through the response‑formatter that enforces the conversational style rules (no all‑caps headers, double line breaks, etc.).

Intent → Category – the NLU layer maps the user’s natural‑language request to one of the top‑level categories (e.g., META_OPTIMIZATION).

Build pipeline – the router reads the list of tool calls under that category, in the exact order shown. Calls are executed sequentially unless marked “parallel” (only quickContext and searchMemories are ever run in parallel).

Collect results – each tool’s JSON response is stored in a temporary context object. If a call fails, the router logs the error and continues with the remaining steps (so the workflow is resilient).

Inject skill – once the final tool in the list finishes, the router adds the associated skill (e.g., meta-optimizer) to the LLM prompt. The skill contains the domain‑specific instructions that turn raw data into a human‑readable answer.

Render response – the LLM produces the answer, which then passes through the response‑formatter (enforces no all‑caps headers, double line breaks, etc.) before being sent back to you.
